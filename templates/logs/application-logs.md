## 1. Purpose and Scope

### 1.1 Purpose

This document establishes the requirements, structure, and governance framework for the collection, retention, and review of **Application Logs** generated by systems classified as National Critical Information Infrastructure (NCII) assets operated by [Organization Name].

Application Logs capture security-relevant events produced by applications, including authentication activities, authorisation decisions, input validation outcomes, application errors, business logic anomalies, and API interactions. These records form a critical component of [Organization Name]'s cybersecurity posture and support incident detection, investigation, and regulatory compliance.

### 1.2 Regulatory Basis

This artifact is mandated under the **Cyber Security Act 2024 (Act 854)**, administered by the **National Cyber Security Agency (NACSA)**. Specifically:

- **Section 18 (s18)** — Imposes obligations on NCII sector leads and NCII entities to maintain security measures, including logging controls, for designated critical information infrastructure.

Compliance with this artifact also supports alignment with:

- **Personal Data Protection Act 2010 (PDPA)** — Logging of access and processing events involving personal data
- **Bank Negara Malaysia (BNM) Risk Management in Technology (RMiT) Policy Document** — Operational and security log management requirements for financial institutions
- **ISO/IEC 27001:2022** — Annex A Control 8.15 (Logging) and Control 8.17 (Clock Synchronisation)

### 1.3 Scope

This policy applies to:

- All applications designated as NCII systems or supporting NCII systems operated by [Organization Name]
- Internal and externally facing applications processing sensitive, regulated, or mission-critical data
- Third-party or vendor-operated applications hosted within or connected to [Organization Name]'s NCII environment
- Application programming interfaces (APIs) exposed by or consumed by in-scope applications

**Out of Scope:**

- Infrastructure-level logs (covered under separate Network and System Log artifacts)
- Non-production environments unless processing live or replicated production data

## 4. Input Validation Errors

*Log events where application input validation controls reject or flag user-supplied data. These logs support detection of injection attacks (SQL, XSS, command injection), malformed request payloads, and fuzzing activity.*

### 4.1 Log Fields — Input Validation Events

| Field | Description | Example Value |
|---|---|---|
| `timestamp` | Date and time of event (UTC, ISO 8601) | `[Timestamp]` |
| `event_type` | Validation event category | `INPUT_VALIDATION_FAILURE`, `SCHEMA_VIOLATION`, `INJECTION_ATTEMPT_DETECTED` |
| `source_ip` | IP address of the client | `[Source IP Address]` |
| `user_id` | Authenticated user (if applicable) | `[User ID / ANONYMOUS]` |
| `session_id` | Session identifier | `[Session ID]` |
| `endpoint` | Application endpoint or function that received the input | `[Endpoint URI / Function Name]` |
| `parameter_name` | Name of the input field or parameter that failed validation | `[Parameter Name]` |
| `validation_rule` | The validation rule that was triggered | `[Rule ID / Name]` |
| `sanitised_input_sample` | A sanitised or truncated representation of the invalid input (no raw sensitive data) | `[Sanitised Sample]` |
| `http_method` | HTTP method used (where applicable) | `GET`, `POST`, `PUT`, `DELETE` |
| `response_code` | HTTP response code returned to the client | `400`, `422`, `403` |

> **Important:** Raw user input must **never** be logged verbatim if it may contain sensitive data (e.g., passwords, PAN, NRIC). Log sanitised or truncated representations only. Comply with PDPA data minimisation principles.

|---|---|---|
